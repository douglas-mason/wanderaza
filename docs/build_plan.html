Day 1 build plan — week by week milestones for a solo bootstrapped travel platform MVP

Literal day 1 — the first 4 commands you run
pnpm create next-app@latest apps/web --typescript --tailwind --app pnpm create fastify apps/api --lang=typescript pnpm dlx shadcn@latest init git init && git remote add origin [your repo]

Before writing a single feature: get both apps running locally, talking to each other, and deployed (Vercel for web, Railway for API). A working CI/CD pipeline on day 1 means every evening session ends with something shipped.

Week 1 Scaffold + infrastructure Goal: everything deployed, nothing useful yet
Monorepo setup infra

pnpm workspaces with /apps/web (Next.js), /apps/api (Fastify), /packages/types (shared TS). Get both apps running and talking to each other locally.

Deploy pipeline day 1 infra

Vercel for Next.js (free). Railway for API + Postgres + Redis ($5–10/mo). Get a real URL immediately — psychological momentum matters for solo projects.

Auth with Clerk infra

Install Clerk, protect nothing yet — just get sign-in/sign-up working. You'll gate itinerary saving behind auth later. Don't let auth block product work.

Drizzle + base schema infra

Set up Drizzle ORM, run first migration. Just a users table and a trips table for now. Don't over-design the schema — add tables as you need them.

Week 2 Core search — events by date Goal: search a city + dates, see real events
Ticketmaster API integration data

Hit the Discovery API with city + date range, return events. This is your hero feature — the thing nobody else does cleanly. Get this working ugly first.

Search UI — city + date picker product

Simple form: destination input (text for now, geocode later), check-in / check-out date pickers, submit. shadcn Calendar component handles the date picker.

Event results list product

Display event name, date, venue, category, price range. Card grid layout. No map yet — list first. Cache Ticketmaster responses in Redis with a 2-hour TTL.

Week 3 Itinerary builder — the core loop Goal: add an event to a trip, see it on a timeline
"Add to trip" action product

Button on each event card that adds it to an active trip. If not logged in, prompt to sign in (Clerk). This is the core interaction — get it feeling fast.

Trip timeline view product

Day-by-day view of the trip with added items. No drag-and-drop yet — just a list per day. Drag-and-drop is a nice-to-have; a working timeline is the requirement.

Shareable trip link ship

Generate a public /trip/[id] URL. Anyone with the link can view the itinerary — no account needed to view, account needed to create. This is your first viral loop.

Week 4 Restaurants + map view Goal: a complete trip has events + dinner options on a map
Google Places — restaurant search data

Nearby restaurant search for the destination city. Store results in Postgres after first fetch — don't re-call Google Places for the same city repeatedly.

Mapbox map view product

Show all trip items (events + restaurants) as pins on a map alongside the list. react-map-gl makes this straightforward. This is the moment the product feels real.

Send to a real person ship

Build a trip for a real destination you know. Send the shareable link to 3–5 friends or family. Watch how they interact with it. This feedback is worth more than a month of solo building.

End of month 1 milestone
A real person can plan a real trip
Search a city + dates → see events → add to itinerary → add restaurants → share a link. That's the loop. Everything after this is refinement and expansion.
